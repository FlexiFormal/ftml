Introduction

Recognizing the discrepancy between formal and informal mathematics as well as formal and informal mathematical practice, the flexiformalist program [Koh13] calls for annotating informal, human-oriented documents with partially formalized (and modular, reusable, shareable, distibuted, etc.) content commons (a.k.a. a domain model). This promises to mitigate the difficulties and required expertise associated with fully formalizing mathematical results, while retaining many of the advantages of formality; most notably, still allowing for most useful software support (apart from full verification), such as disambiguation, interlinking, lookup, documentation, semantic search, etc.
Naturally though, fulfilling this promise in practice requires comprehensive software support for the entire producer-to-consumer pipeline of flexiformal documents, spanning an entire ecosystem of associated activities starting with
1. authoring documents,
2. authoring/generating/extracting the content commons,
3. curating the domain model FAIRly (findably, accessibly, interoperably, reusably), and
4. viewing/sharing/publishing the flexiformal documents, with
5. the promised added value software services integrated.
The end result of the process are active documents that benefit readers by making the knowledge contained in them (and prerequisite knowledge) more explicit, accessible, and easier and faster to learn.
All of the steps above have been realized (usually) in relative isolation, with the result that following the entire pipeline from authoring to viewing documents has been a daunting process, involving multiple partially instable systems at precise versions, with each step being error prone, difficult to set up, maintain and debug.

The primary shortcomings of the Mmt-centered previous incarnation of our flexiformal document ecosystem were the result of a lack of support for documents and document fragments, which [Ber+23] calls the formulation model. Domain Model

For the domain model, we largely follow Mmt/OMDoc, and as such OpenMath, with the notable distinction that we provide first-class support for structures as record( type)s, conservative extensions, and opaque (flexiformal) expressions. All three of these have existed previously in Mmt, encoded as plain OpenMath, and as such are not described in detail here.
The domain model consists of modules, which are named (by a Module URI) lists of the following declarations:
Nested

Symbols
